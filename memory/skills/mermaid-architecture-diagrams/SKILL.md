# Mermaid architecture diagrams

## Use for

- system context
- service ownership
- event choreography
- sequence flows
- deployment or runtime views
- documentation updates after architecture changes

## Rules

- Use repository-accurate names.
- Show ownership and direction clearly.
- Keep diagrams source-controlled and editable.
- Pair diagram changes with nearby docs changes.

## Read first

- `/memory/rules/05-diagrams-and-docs.md`
- `/memory/policies/07-documentation-and-traceability.md`

## Output

Provide:
- the Mermaid diagram,
- where it should live,
- any doc sections that should be updated with it.
